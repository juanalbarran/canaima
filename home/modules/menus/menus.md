# Menu System

Two menu backends — **wofi** (GUI, Wayland-native) and **bemenu** (lightweight dmenu-like) — configured as Nix modules. The active backend is set once per profile via `hostSpec.menu`; all launcher scripts consume it via that variable so swapping backends requires changing a single line.

## Directory Layout

```
home/modules/menus/
├── wofi/
│   ├── default.nix           # programs.wofi + xdg.configFile for all conf files
│   ├── style.css             # Static reference/draft (NOT deployed at runtime — see Style section)
│   ├── config-menu.conf      # Used by: system-menu, power-menu (small menus, no search)
│   ├── bookmarks-menu.conf   # Used by: bookmarks second-level (wider, search visible)
│   ├── prebookmarks-menu.conf # Used by: bookmarks first-level / category picker
│   └── projects-menu.conf    # Used by: projects (wide, search visible)
└── bemenu/
    └── default.nix           # programs.bemenu with full color/font/layout settings
```

## Backends

### wofi

Wayland-native graphical menu. Used by `playa-el-yaque` (Ubuntu work laptop).

`wofi/default.nix` does two things:

1. Enables `programs.wofi` with global defaults (`allow_images`, `insensitive`, `run-cache_file=/dev/null`).
2. Deploys all `.conf` files to `~/.config/wofi/` and wires the runtime style via a symlink to the theme cache (see Style section).

Each script that needs a menu opens wofi with a specific `.conf` file tuned to its use case:

| Config file              | Width | Height | Search visible | Used by                            |
|--------------------------|-------|--------|----------------|------------------------------------|
| `config-menu.conf`       | 250   | 300    | No             | `system-menu`, `power-menu`        |
| `prebookmarks-menu.conf` | 300   | 250    | No             | `bookmarks` — category picker      |
| `bookmarks-menu.conf`    | 500   | 400    | Yes            | `bookmarks` — links list           |
| `projects-menu.conf`     | 400   | 300    | Yes            | `projects`, `projects-ctwo`        |

All configs share: `location=center`, `hide_scroll=true`, `insensitive=true`, vim-style keys (`Ctrl-p`/`Ctrl-n`).

### bemenu

Lightweight dmenu-compatible launcher. Used by `playa-el-agua` (NixOS main laptop) and `playa-caribe`.

Configured entirely in `bemenu/default.nix` via `programs.bemenu` (Home Manager option). Key settings:

- 10 lines visible, `wrap=true`, centered, 30% screen width
- Font: JetBrains Mono 12
- Colors: dark `#131314` background, `#cdd6f4` text, `#005577` highlight/border

## hostSpec.menu — How Backends Are Selected

`hostSpec.menu` holds the full command string (including flags) that gets baked into every script wrapper at `home-manager switch` time. Scripts never hard-code a menu program.

| Profile          | `hostSpec.menu` value                                              | Active backend |
|------------------|--------------------------------------------------------------------|----------------|
| `playa-el-agua`  | `"bemenu"`                                                         | bemenu         |
| `playa-el-yaque` | `"wofi --conf $HOME/.config/wofi/projects-menu.conf --prompt 'Projects:'"` | wofi |
| `playa-caribe`   | `"bemenu"`                                                         | bemenu         |

> **Note:** The `playa-el-yaque` menu value includes wofi flags that were originally written for the `projects` script. These flags get passed to every script (system-menu, bookmarks, power-menu…), which works because each script either overrides them via its own `run_menu()` wrapper or ignores extra flags. Ideally this should be just `"wofi --dmenu"` and each script should own its conf path.

## Wofi Style

### Runtime style (what you actually see)

`wofi/default.nix` deploys `~/.config/wofi/style.css` as a symlink:

```
~/.config/wofi/style.css → ~/.cache/style/wofi.css
```

`~/.cache/style/wofi.css` is generated by the theme system at `toggle-theme` time from `home/modules/ui/themes/templates/wofi.nix`. This means the wofi colors automatically track dark/light mode alongside every other app.

The template uses Base16 palette slots:

| CSS element          | Base16 slot | Role                        |
|----------------------|-------------|-----------------------------|
| Window background    | `base00`    | Background                  |
| Window border        | `base0D`    | Accent (blue)               |
| Text                 | `base05`    | Main text                   |
| Selected background  | `base0D`    | Accent                      |
| Selected text        | `base00`    | Contrast text on selection  |
| Input field          | `base00`    | Background (matches window) |

### The static `wofi/style.css` in the repo

This file is **not deployed at runtime** — the symlink above takes precedence. It is kept as a hardcoded-color reference draft (Catppuccin values, `#input { display: none }` to hide the search bar). Changes here have no effect until they are ported into `themes/templates/wofi.nix`.

## Towards an Omarchy-style Look

Omarchy (DHH's Arch Linux setup) used wofi until v1.5 before switching to Walker. Its wofi style is noticeably more polished than the current setup. The key visual differences and what it would take to close them:

### What omarchy does differently

| Property            | Current setup                       | Omarchy-style target                   |
|---------------------|-------------------------------------|----------------------------------------|
| Corner radius       | `0px` — sharp corners everywhere    | `~10px` on window + entries            |
| Font family         | JetBrains Mono (monospace)          | Sans-serif (Roboto, or Inter)          |
| Font size           | `14px` fixed                        | `1rem` / `1.2rem` for input            |
| Padding             | `5px` margins only                  | `0.5rem`–`1rem` all around             |
| Input field         | Shown but plain / sometimes hidden  | Prominent, larger font, top of window  |
| Entry row height    | Default (no explicit padding)       | Explicit padding (`0.25rem 0.5rem`)    |
| Border              | 2px solid accent                    | Thinner or no border; rely on radius   |
| Hover state         | Only `:selected` styled             | Separate `:hover` style (lighter tint) |

### Changes needed in `themes/templates/wofi.nix`

Since the runtime style is generated by the theme template, all visual changes must go into that file. A closer-to-omarchy version would look roughly like:

```css
window {
  margin: 0px;
  border: 1px solid <base0D>;
  border-radius: 10px;
  background-color: <base00>;
  font-family: "Inter", "Roboto", sans-serif;
  font-size: 1rem;
}

#input {
  margin: 0.5rem;
  padding: 0.5rem;
  border: none;
  border-radius: 6px;
  background-color: <base01>;
  color: <base05>;
  font-size: 1.1rem;
  outline: none;
}

#entry {
  padding: 0.25rem 0.5rem;
  border-radius: 6px;
}

#entry:hover {
  background-color: <base02>;
}

#entry:selected {
  background-color: <base0D>;
  border-radius: 6px;
  outline: none;
}

#text {
  margin: 0.25rem 0.5rem;
  color: <base05>;
}
```

> `base01` is the "lighter background" slot — a good subtle contrast for the input field. `base02` is the "selection background" slot — right for hover.

### Font note

Switching to `"Inter"` or `"Roboto"` for menus while keeping JetBrains Mono for terminals gives the launcher a softer, more launcher-like feel. Requires those fonts to be in `home.packages`; both are available in nixpkgs as `pkgs.inter` / `pkgs.roboto`.

## Future Improvements

- **Unify `playa-el-yaque` menu value**: change `hostSpec.menu` for that profile to just `"wofi --dmenu"` and let each script manage its own `--conf` path. The current value with embedded flags is fragile and confusing.
- **Theme-aware bemenu**: `bemenu/default.nix` uses hardcoded hex colors; these should be read from the same `~/.cache/style/` mechanism the theme system uses so bemenu also tracks dark/light switches. This would require a new `themes/templates/bemenu.nix` and a symlink approach analogous to wofi.
- **Retire the static `wofi/style.css`**: once the omarchy-style improvements are in `wofi.nix`, delete the static file so there is no confusion about which file is the source of truth.
- **Walker migration**: omarchy moved to [Walker](https://walkerlauncher.com/) in v1.6 for a faster, more feature-rich launcher (fuzzy search, plugins, better theming). Worth evaluating as a replacement for wofi on `playa-el-yaque`.
- **Single conf file**: the four separate `.conf` files differ only in `width`, `height`, and `hide_search`. A single conf with overrides passed as CLI flags (`--width`, `--height`, `--show-search`) per call site would halve the config surface.
- **Consistent key navigation**: bemenu and wofi both support `Ctrl-p`/`Ctrl-n` but the `programs.bemenu` Nix option doesn't expose key bindings — they'd need to be set via `home.sessionVariables`. Currently only wofi has explicit vim-style nav configured.
